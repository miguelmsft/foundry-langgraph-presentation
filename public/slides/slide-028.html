The Tools

🔍
list_tables
Discover available tables
Safety: Read-only connection
📋
get_table_schema
Get column names, types, row count
Safety: Table allowlist validation
📊
query_database
Execute SQL queries
Safety: SELECT-only, keyword blocking, 100-row limit

Safety layers in query_database:

# 1. Only SELECT allowed if not normalized.startswith("SELECT"): ... # 2. Block dangerous keywords dangerous = {"INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "CREATE"} # 3. Read-only database connection conn = sqlite3.connect("file:data/logistics.db?mode=ro", uri=True) # 4. Truncate results to 100 rows rows = cursor.fetchall()[:100]
We have three tools, each with a clear role. list_tables for discovery, get_table_schema for understanding structure, and query_database for execution. The safety features are layered: get_table_schema enforces a table allowlist. query_database uses read-only mode plus dangerous keyword blocking — the database is opened in read-only mode at the SQLite level, we validate that queries start with SELECT, we block dangerous keywords, and we truncate results to 100 rows. Defense-in-depth is important when you're letting an LLM write SQL.
